### A1. [MAJOR] verifyAndCopyAttachments and all frontend storage cleanup helpers have zero test coverage — the gate's claim they are 'exercised indirectly via the helper contracts' is false
- **Lens:** test-coverage  ·  **Where:** `frontend/lib/storage.ts` : 458-577 (verifyAndCopyAttachments), 291-309 (bestEffortRemove), 426-435 (removeStagedFiles), 586-598 (removeRunSources)
- No test anywhere in the repo imports anything from frontend/lib/storage.ts (verified by grep: only routes and the peer-review doc reference these symbols). This function contains the runtime implementation of the Phase-2 gate's two highest-severity integrated fixes: the post-copy TOCTOU size re-verify (Gemini BLOCKING #1, lines 555-574) and the partial-copy rollback (Gemini MAJOR #3, lines 529-537), plus verify-all-before-copy ordering (508-525), bestEffortRemove's resolved-{error} handling (Codex MINOR #3), and removeStagedFiles (Codex MAJOR #2 first half). The phase-2 merge-gate synthesis states these are 'exercised indirectly via the helper contracts' — that is materially inaccurate: the tested pure helpers (buildCopyPlan/stripOrigin/partitionByOrigin) never execute the copy loop, the rollback, or the re-verify. A regression in any of these branches (inverted size comparison, deleted re-verify block, rollback removing wrong paths, copy-before-verify reorder) passes the full 364-test suite silently. The function is untestable as written because it hard-codes getSupabase() — in contrast, the agent-side downloadAttachments(sb, ...) was given the injectable StorageDownloaderLike interface specifically so its equivalent branches COULD be unit-tested (and all are). The asymmetry is unjustified: the frontend orchestration is the same risk class (SECURITY/DATA labeled) and the same mockability shape.
- **Recommendation:** Mirror the agent pattern: add an injectable client parameter (or StorageOrchestratorLike structural type) to verifyAndCopyAttachments/bestEffortRemove/removeStagedFiles/removeRunSources, then pin: (1) not-found and size-mismatch return 400 BEFORE any copy() call (assert mock copy never invoked); (2) copy failure mid-plan rolls back exactly the copied-so-far toPaths and returns 500; (3) post-copy destination list mismatch removes ALL copies and returns 400; (4) destination-list error removes copies and returns 500; (5) bestEffortRemove logs a RESOLVED {error} and never throws. Also correct the test-coverage statement in the phase-2 synthesis doc so the next audit doesn't inherit the inaccurate claim.

### A2. [MAJOR] No guard against in-flight uploads at Next/Submit — submission races upload completion and silently drops the attachment
- **Lens:** ui-state  ·  **Where:** `frontend/hooks/useNewResearchForm.ts (also frontend/components/new-research/StepTopic.tsx, frontend/components/new-research/StepReview.tsx)` : useNewResearchForm.ts:329-370 (goNext), 381-425 (onSubmit); StepTopic.tsx:59, 134-137; StepReview.tsx:303-307

(StepTopic.tsx:195) fires the form.watch subscription (useNewResearchForm.ts:125-135), which re-writes the just-cleared draft to sessionStorage — the fully-submitted form resurrects as a 'draft' (including the orphaned staged attachment) on the next /new visit in that tab.
- **Recommendation:** Lift an isUploading flag (or the count of in-flight uploads) out of StepTopic into the form hook (e.g. a transient form field or shared state). Block or warn on goNext from the topic step while uploads are pending, and hard-disable the StepReview submit button with an 'uploads in progress' message. Unsubscribe the form.watch persistence (or set a 'submitted' flag checked in the watch handler) before clearing sessionStorage in onSubmit so a late upload completion cannot resurrect the draft.

### A3. [MAJOR] Removing an in-flight upload does not cancel it — the 'removed' file is silently appended to the form and ships with the run
- **Lens:** ui-state  ·  **Where:** `frontend/components/new-research/StepTopic.tsx` : 225-249 (removeItem early-return), 186-201 (post-PUT form append); no AbortController anywhere in the file
- removeItem for a status 'uploading' row only filters the UI row out of uiItems (StepTopic.tsx:227-229) — there is no AbortController and the mint+PUT continuation in uploadFile keeps running. On completion it unconditionally executes setValue('attachments', [...current, item]) (line 195); the subsequent setUiItems map (line 197-201) is a no-op because the tempId row is gone, so no chip reappears. Net effect: the user clicks X on a file mid-upload, the row visibly disappears, and the file is re-attached invisibly moments later — it appears again only on the Review step's Attached Files list (or in the small 'N files attached' count). For the canonical wrong-file scenario (user drops a confidential PDF, immediately removes it), the file still gets copied into <orgId>/<slug>/sources/, digested, and fed into the LLM prompt chain — the explicit removal action is silently overridden. The comment at lines 223-224 ('For error/uploading rows (no storedName) just drop the UI row') shows the author assumed an uploading row would never materialize into form state, which the code contradicts.
- **Recommendation:** Give each uploadFile an AbortController stored on the UiAttachment; removeItem on an uploading row should abort() it. As a belt-and-suspenders backstop, have the success path check that the tempId row still exists in uiItems (functional-state read) before appending to the form, and best-effort DELETE the staged object if the row was removed mid-flight.

### A5. [MAJOR] All-skipped attachments are silently invisible to the user — and common legacy text encodings (Windows-1252, UTF-16) are realistic legit triggers
- **Lens:** worker-correctness  ·  **Where:** `c:/Users/ceo/Documents/AI Training/Anti Gravity/Dynamic Research/agent/lib/attachments.ts` : 87-95 (sniff NUL + strict-UTF-8 checks); also agent/executor.ts:873-888 (attachmentsBlock is the empty string when downloaded.length===0)
- The gates accepted SKIP-AND-RECORD, but the 'record' is only the ephemeral workdir manifest (never uploaded — uploadOutputs sources only Projects/<slug>/, and upload-set excludes the sources/ subdir) plus worker.log. Nothing durable or user-visible captures the outcome: the completion email (agent/lib/notify.ts — zero attachment references), the DB row (no updateJob field carries skip info), and the clone/gallery manifest route (which serves the ORIGINAL attachments list from research_queue) all present the attachments as if they were used. When downloaded===0 but skipped>0 (fail-closed list error, or every file failing sniff), buildPrompt emits NO attachment text at all — the orchestrator runs completely sourceless, the run completes, and the user gets a normal 'completed' email with no indication their files were ignored. This is not just the accepted forged-row/MZ-edge path: a .txt/.md saved as Windows-1252 (smart quotes 0x93/0x94 are invalid UTF-8 → 'not valid UTF-8' skip) or UTF-16 (NUL bytes → 'contains NUL bytes' skip) is extremely common on Windows, passes ALL Phase-2 frontend validation (extension+MIME+size only — the signed-URL PUT bypasses server content checks by design, per the module's own header comment), and is then silently dropped by the worker. Only the 'MZ-prefix legit text' false positive was listed as an accepted residual at the gate; the legacy-encoding class and the zero-surfacing of the all-skipped case were not.
- **Recommendation:** Two halves: (1) Surface the outcome durably — thread attachmentsResult into notifyTerminal/sendCompletionEmail (a one-line 'N of M attachments could not be used: <reasons>' note) and/or persist a skip summary via updateJob so the gallery can show it; when downloaded===0 && skipped>0, still emit a short non-fenced notice in the prompt so the orchestrator can state in the report that user sources were submitted but unavailable. (2) Shrink the trigger surface — for text/plain + text/markdown, transcode-or-accept legacy single-byte encodings (e.g., fall back to windows-1252 decode when strict UTF-8 fails, re-encode to UTF-8 before writing) or validate encoding client-side at upload time where the user can react.

### A7. [MINOR] Total-size cap enforced only AFTER the file is fully downloaded and buffered — incomplete application of the G2 pre-download gate
- **Lens:** resource-cost  ·  **Where:** `agent/lib/attachments.ts` : 347-353 (vs. pre-download gate at 309-326)
- The G2 fix moved the PER-FILE size gate before download(): the sources/ listing is fetched once and listedSize must exactly equal the declared sizeBytes before any bytes are pulled (lines 309-326). But the TOTAL cap (max_total_bytes, 40MB) is still checked at line 347 — AFTER download() + arrayBuffer() have buffered the whole object. Because the exact size of every file is already known and verified pre-download, files that can only ever be skipped by the total cap are still fully downloaded and buffered first. Worst case (5 valid 15MB files = 75MB declared > 40MB cap): files 3-5 are each pulled (~45MB of unnecessary storage egress, repeated 15MB RAM buffers, wasted wall-clock inside the job) and then discarded. Reaching this path requires a forged row (legit rows are bounded to 40MB total by attachmentsArraySchema at submit), which is the same accepted defense-in-depth threat model the per-file gate was built for — so the guard pattern is applied inconsistently within its own scenario. No OOM risk (bounded at 15MB per buffer), purely wasted egress/RAM/time.
- **Recommendation:** Move the running-total check before the download() call: since `listedSize === meta.sizeBytes` is verified at line 320, check `totalBytes + meta.sizeBytes > ATTACHMENTS.max_total_bytes` immediately after that gate and skip without downloading. Keep the post-download check as belt-and-braces if desired.

### A8. [MINOR] Daily staging sweep runs inline on the worker poll chain with no time budget — job claiming is blocked for the sweep's full (prefix-count-unbounded) duration
- **Lens:** resource-cost  ·  **Where:** `agent/worker.ts` : 247 (poll), 226-232 (sequential pollAndContinue chain); agent/lib/staging-sweep.ts:160-191
- poll() awaits maybeRunStagingSweep() on idle ticks, and schedulePoll() only fires after pollAndContinue() completes — so the next claimJob() cannot run until the sweep finishes. The comment says the sweep 'never delays a claimed job', which is true, but it delays the NEXT claim by the full sweep duration. Per-prefix work is bounded (MAX_PAGES=20 → 20k entries), but the NUMBER of prefixes is not: the sweep issues 1 sequential storage list() call per org folder plus 1 per draft folder plus delete batches (staging-sweep.ts:160-191, all sequential awaits, no overall time cap and no mid-sweep queue re-check). A few thousand abandoned drafts (multi-org growth, or one authenticated client minting fresh draftIds at the rate-limit ceiling of ~480/day/IP across several IPs) turns the daily idle tick into a multi-minute-to-tens-of-minutes block on job pickup. The 5-minute cron cannot spawn relief: the worker IS running and ensureSingleton (worker.ts:56-83) makes a second instance exit 2. A job submitted just as the sweep starts waits the entire sweep.
- **Recommendation:** Give the sweep an overall time budget (e.g. stop listing after N seconds or M prefixes and set truncated, resuming next day), or check for a pending job between org iterations and bail early. Alternatively run the sweep detached from the poll chain (fire-and-forget with its own guard flag) so claimJob latency is never coupled to staging volume.

### A9. [MINOR] 24h sweep gate fails OPEN: marker written only after a completed sweep — mid-sweep worker death or persistent marker-write failure causes a full bucket scan on every 30s idle tick
- **Lens:** resource-cost  ·  **Where:** `agent/lib/staging-sweep.ts` : 284-290 (marker write), 251-258 (marker read)
- The in-code comment claims the marker is 'written after every attempt — success OR failure — so a failing sweep retries tomorrow instead of hammering storage on every 30s poll'. That only holds when the marker WRITE succeeds and the process survives to reach it. Two fail-open paths: (1) the marker is written only AFTER sweepStagingUploads() returns — if the worker is killed/crashes mid-sweep (the documented TaskStop-orphan and cron-respawn environment makes worker terminations routine), the next worker's first idle tick re-runs the FULL sweep from offset zero, repeating all list calls and partially-done delete batches; with a long sweep (see the blocking finding) this can become a restart-storm pattern. (2) If the writeFile at line 285 fails persistently (disk full, EPERM on the marker path mid-life — startup cwd-writability is only proven once via .worker.pid), the error is logged non-fatal and the next tick reads a missing/corrupt marker → 'due now' → full multi-org storage scan every 30 seconds indefinitely: continuous Supabase list-API hammering and log spam, exactly the failure the marker exists to prevent.
- **Recommendation:** Write the marker (or an in-progress sentinel) BEFORE starting the sweep and finalize it after, so a crashed or long sweep cannot re-trigger immediately; on marker-write failure, fall back to an in-memory last-run timestamp for the lifetime of the current process so a single worker can never sweep more than once per interval.

### A12. [MINOR] Persistent marker-write failure degrades the 24h gate to a full sweep every 30s idle tick — the exact hammering the marker comment claims to prevent
- **Lens:** sweep-correctness  ·  **Where:** `c:/Users/ceo/Documents/AI Training/Anti Gravity/Dynamic Research/agent/lib/staging-sweep.ts` : 236-240, 284-290 (and agent/worker.ts:243-248)
- The doc comment states 'The marker is written after every attempt — success OR failure — so a failing sweep retries tomorrow instead of hammering storage on every 30s poll.' That holds for SWEEP failures (errors accumulate in stats and the marker still gets written), but not for MARKER-WRITE failures: fs.writeFile(markerPath, ...).catch(...) logs 'non-fatal' and returns {ran:true}. On the next idle tick (30s later) the marker is still missing/stale, the gate reads 'due', and the ENTIRE sweep re-runs — root + per-org + per-draft storage listings plus delete attempts — every 30 seconds for as long as the write keeps failing (read-only cwd, EPERM, disk full). Disk-full is exactly the condition under which the write fails persistently while network storage calls keep succeeding, producing sustained Supabase list-API churn and worker.log spam from a single long-running worker process. The worker process itself persists across ticks (PID rotates only on exit), so an in-memory fallback would fully bound this within a process lifetime; nothing in the code uses one.
- **Recommendation:** Keep a module-level (in-memory) lastRunMs in staging-sweep.ts, set after every attempt, and treat the gate as max(markerFile, inMemory). The file marker still covers cross-respawn pacing; the in-memory value covers the marker-unwritable case for the life of the process. One-line state + one-line check; add a unit test that injects an unwritable markerPath and asserts the second call within 24h returns {ran:false}.

### A15. [MINOR] MAX_PAGES truncation branch (integrated Gemini G3 fix) has no test — only truncated===false is ever asserted
