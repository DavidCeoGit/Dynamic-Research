## ⚠️ Verification corrections (don't act on the raw audit for these)

- **[REFUTED] claim_next_job non-atomic race** (`frontend/app/api/queue/claim/route.ts:25-52`, audit said HIGH). The RPC is absent, but the fallback `UPDATE … WHERE id=X AND status='pending'` is **atomic in Postgres** — two agents that both SELECT the same job serialize on the UPDATE; the loser matches 0 rows and returns 204. No double-claim. **Not a bug.** (Optional polish: actually create the `claim_next_job` RPC with `FOR UPDATE SKIP LOCKED` to remove the wasted round-trip + the misleading comment — LOW.)
- **[SEVERITY ↓ to MEDIUM] agent auth** (`claim/route.ts:17`, `[id]/route.ts:61`). Real hardening, not HIGH: (a) use `crypto.timingSafeEqual` instead of `!==`, (b) fail-loud startup validation that `AGENT_SECRET_KEY` is set, (c) optional per-IP failure rate-limit. The key is a long shared secret on an agent-only route; risk is modest.
- **[KNOWN/ACCEPTED] in-memory rate limiter** (`frontend/lib/rate-limit.ts`). The code itself documents the Vercel-distributed gap + the S52 Upstash-Redis follow-on. `maxOutputTokens` caps already bound per-call cost. Not new. → backlog (Upstash) when the unauth routes see real abuse.
