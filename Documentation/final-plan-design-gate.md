# Final-Plan DESIGN-Gate — Pre-Worker-Spawn Multi-Reviewer Quality Bar

**Status:** DESIGN v3 (Codex DESIGN-gate findings integrated after Codex BLOCK on v2; pending Codex sequential QA pass)
**Author:** Claude Opus 4.7 (1M context)
**Date:** 2026-05-26 (S57 / pre-S58)
**Scope:** Add a sequential Claude → Gemini → Codex review gate to the worker pipeline, fired AFTER queue claim and BEFORE the expensive `claude -p` spawn. The artifact reviewed is a NEW structured research plan synthesized from the user's manifest. Reject/iterate before spending $10-30 + 30-60 min on a misguided run.

**Out of scope (per S57 conversation):** multi-reviewer at the topic-intake or question-generation stages. Those keep single-model Claude (with optional self-critique pass). MRPF cost asymmetry doesn't hold for cheap recoverable interactive form steps.

---

## 1. Problem statement

The current pipeline spends ~$10-30 in tokens + 30-60 min wall-clock per `claude -p` worker spawn. Bad inputs upstream (vague topic, mis-scoped questions, wrong vendor universe) waste the full run because nothing validates the planned research approach before the expensive synthesis fires. The only existing post-spawn quality gate is `verifyPipelineCompletion()` (S34 + S57 Bug 49 fix), which catches pipeline collapse, NOT plan misalignment.

Concrete examples of plans that the current system would happily execute and waste money on:
- Topic asks about "Tesla solar adoption" but extracted context tracks "Tesla EV market share" — single-pivot drift, no recovery
- Question answers narrow to a regional sub-market but the plan still uses the broad-market vendor list
- Persona is "CMO" but the plan's scoring rubric weights engineering-depth metrics
- Studio-product selection includes video but the topic has no visual narrative arc (Veo wastes runtime per [[feedback_veo3_cinematic_hard_runtime_cap]])

MRPF dogfooding has shown Gemini's whole-artifact read + Codex's code-grounded pass catch failure modes Claude alone misses. Apply the same discipline to the plan that the worker is about to execute.

### Proposed flow (incremental insertion — NO change to research_queue.status enum):
```
User form → /api/queue insert (status='pending', plan_review_status='pending')
  → worker.claimJob() (status → 'running' as before)
    → workdir setup + .claude/sandbox-allowlist write     (existing)

         APPROVED → plan_review_status='approved'
         REQUEST_CHANGES → plan_review_status='request_changes' (user-input needed; status stays 'running' until user responds OR cancels)
         BLOCKED (plan quality)  → plan_review_status='blocked' + status='failed' (terminal)
         SYSTEM_BLOCKED (infra) → plan_review_status='system_blocked' (job released for retry; status RESET to 'pending' with plan_review_next_attempt_at = now() + delay)
    → if plan_review_status='approved': proceed
    → studio_only branch decision (existing) — does plan review apply to studio_only? See §3.1 below.
    → dry-run check                                       (existing)
    → claude-prompt.md write                              (existing; also embeds plan_json)
    → spawn claude -p with plan_json as additional input  (existing path + plan)
```

**research_queue.status enum is UNCHANGED.** Existing values `pending | running | completed | failed | cancelled` continue to govern queue lifecycle. The NEW dimension is `plan_review_status`, which the UI reads alongside status to render the right badge.

### §3.1 — studio_only bypass decision (NEW, Codex MAJOR 1)
studio_only mode regenerates NLM Studio products against an existing parent notebook — the manifest contains exact product selections + parent notebook ID and is by-construction much smaller than a fresh-research manifest. Two options:
- **Option A: studio_only bypasses plan review.** Cost: $0 review per studio_only job. Risk: a manifest with stale parent notebook ID or wrong product selection still spawns claude -p. But studio_only jobs are typically retries — the user already approved the original plan.
- **Option B: studio_only goes through abbreviated plan review.** Synthesizer skips topic/persona/vendor sections (they're inherited); reviewers focus on whether the requested product set makes sense for the parent. Cost: ~$3-5 per call (~50% of full review).

**Recommend Option A for Phase 1 MVP.** Phase 3 telemetry decides if studio_only error rate justifies Option B.

### New module: `agent/lib/plan-reviewer.ts`

  plan: ResearchPlan,
  manifest: JobManifest,

### `research_queue` table additions (Codex CRITICAL 1 + CRITICAL 2 fix):

**research_queue.status enum is UNCHANGED.** Existing values `pending | running | completed | failed | cancelled` continue to govern queue lifecycle. New columns add a parallel dimension for plan-review state + retry tracking.

```sql
ALTER TABLE public.research_queue
  ADD COLUMN plan_json jsonb,
  ADD COLUMN plan_review_status text NOT NULL DEFAULT 'pending'
    CHECK (plan_review_status IN (
      'pending',         -- never reviewed yet
      'reviewing',       -- review in progress
      'approved',        -- approved for spawn
      'request_changes', -- user-input needed (NEVER for system failure)
      'blocked',         -- plan-quality reject (terminal — status set to 'failed')
      'system_blocked'   -- infra failure; auto-retried (status reset to 'pending')
    )),
  ADD COLUMN plan_review_iterations integer NOT NULL DEFAULT 0,
  ADD COLUMN plan_review_attempts integer NOT NULL DEFAULT 0,      -- system_blocked retry counter
  ADD COLUMN plan_review_next_attempt_at timestamptz,              -- claim predicate field
  ADD COLUMN plan_review_error text;                                -- last error message (truncated 500 chars)
```

### Auto-retry mechanics (Codex CRITICAL 2 — fully specified)

The same worker does NOT hold the job during system-failure retry. On SYSTEM_BLOCKED:
1. Worker writes `plan_review_status='system_blocked'` + increments `plan_review_attempts` + sets `plan_review_error` + sets `plan_review_next_attempt_at = now() + delay`
2. Worker resets `status='pending'` (releases the job back to the queue)
3. Worker moves on to next pending job (no in-process retry — avoids head-of-line blocking)
4. The claim predicate is extended to also pick up retryable jobs in `plan_review_status='system_blocked'`:

```sql
-- Modified claim predicate (replaces existing in frontend/app/api/queue/claim/route.ts:30):
-- (Codex v3 QA fix: single AND-clause; system_blocked rows MUST honor next_attempt_at delay)
UPDATE research_queue SET status='running', claimed_at=now()
WHERE id = (
  SELECT id FROM research_queue
  WHERE status='pending'
    AND (
      plan_review_status <> 'system_blocked'
      OR plan_review_next_attempt_at <= now()
    )
  ORDER BY plan_review_next_attempt_at NULLS FIRST, created_at ASC
  LIMIT 1
  FOR UPDATE SKIP LOCKED
)
RETURNING ...
```

**Retry schedule:** initial delay 5 min, then exponential — delays 5m, 15m, 45m, 2h15m, 6h45m, 20h15m (max 6 attempts). Total max wait ~30 hours then operator-alert via Resend email + plan_review_status='blocked' (terminal).

**Concurrency guarantee:** the `FOR UPDATE SKIP LOCKED` clause + the `plan_review_next_attempt_at` predicate prevent two workers from racing on the same retry. Existing claim semantics in `frontend/app/api/queue/claim/route.ts` already use this pattern.

### Fallback behavior (CRITICAL — these MUST be defined before merge)

**Principle (Gemini v1 CRITICAL 1):** system-side failures NEVER blame the user. `REQUEST_CHANGES` is for plan-quality problems with the user's manifest. `SYSTEM_BLOCKED` is a separate terminal state for infrastructure failures.

- If Gemini API unreachable (5xx, timeout > 120s, rate limit) → retry once with backoff; if still failing, log + proceed with Codex-only review + flag in synthesis as "operating under reduced review" per MRPF severity-mode handling
- If Codex API unreachable → same pattern, Gemini-only
- If BOTH unreachable → mark `plan_review_status='system_blocked'` (per §5 retry schedule: 5m → 15m → 45m → 2h15m → 6h45m → 20h15m, max 6 attempts) with reason "Reviewers unavailable — system issue, not your input. We'll retry automatically." After max attempts, escalate to operator alert
- If review exceeds `REVIEW_TIMEOUT_MS` (default 15 min total) → mark `plan_review_status='system_blocked'` (same §5 retry schedule) with reason "Review timed out — system issue, not your input. We'll retry automatically."
- Under NO circumstance does a system-side failure result in `REQUEST_CHANGES` (which would force the user to "revise" perfectly fine input)

### Cost projection (Gemini v1 MAJOR 2: reframed)

- Average plan review: 1 synth ($1) + 2 review rounds × (Gemini $1.50 + Codex $2) + 2 integration passes ($0.50 each) = **~$8-12 per plan review**
- Adds to existing per-run cost (~$10-30 baseline) → **new per-run total ~$18-42 — this is a guaranteed cost increase, not a break-even**
- The prior version of this doc claimed "if review prevents 1-in-3 wasted spawns → net cost neutral." Gemini correctly flagged this math as flawed: review is a GUARANTEED tax on every run for a PROBABILISTIC savings. If prevention rate is 1-in-3, reviewing 3 runs costs $24-36; catches 1 bad save of $10-30 — net loss. True break-even requires prevention rate > 60% on runs that would otherwise have completed-but-misaligned, which is implausible.

**Justification reframed:**
- This is a **quality-and-safety cost**, not a savings measure
- Pays for: (a) operator confidence the worker isn't burning $30 on wrong-scope runs, (b) caught regressions like Bug 49 surfaced via second-model review, (c) user trust that a "Researching" badge means a vetted plan
- Phase 3 telemetry is the gate that decides whether the cost is justified: if measured prevention-rate × prevented-cost is < review-cost per run (cohort-averaged), the design must contract or sunset
- Acceptable thresholds (decision-makers must agree before launch): are we OK with a 50-80% cost increase per research run in exchange for a quality bar?

## 8. Failure modes + open design questions

### MUST-decide before merge:
1. **Review iteration cap.** Default `MAX_REVIEW_ROUNDS = 2` proposed. Justification: more rounds = more cost, diminishing returns per MRPF S48 dogfood (round 3 added <10% value over rounds 1-2). Pushback?

2. **What counts as APPROVE?** Both reviewers APPROVE? Or APPROVE-WITH-CHANGES from either reviewer where all CRITICAL findings have been integrated? Match MRPF v2.2 disagreement procedure: SECURITY-labeled CRITICAL = blocking; otherwise synthesis can override individual MINOR findings.

3. **Cost-cap circuit breaker.** Hard ceiling per job (e.g. `MAX_REVIEW_COST_CENTS = 500`)? If review-cycle cost exceeds, abort with `review-blocked` + reason. Prevents runaway billing.

4. **Cold-start latency tax.** Currently 0 cost on review (because no review). New baseline = ~5-10 min review BEFORE the existing 30-60 min run. User waits MORE total time even on approved plans. Is this acceptable trade for quality? — alternative: parallel preview of "we'd ship in X min if approved" while review runs.

5. **Skip-review mechanism (Codex v2 MINOR 2 — Phase 2 alignment).** Per-job `skip_review` field on the queue submission, scoped to dev workspace + internal test org (not production org). Recorded as such in telemetry. Narrower than a per-user trusted_bypass flag; no expansion of production permissions surface; auditable per-job. **Phase 2 §9 below is aligned with this single mechanism — NO per-user trusted-bypass flag is in scope** (removes the v2 contradiction Codex flagged).

6. **Plan format versioning.** `schema_version: 1` lets us evolve the plan schema without breaking historical reviews. Future v2 might add e.g. `evaluation_constraints` block.

### KNOWN failure modes to test:
- Gemini API rate-limit during business hours
- Codex API auth-rotation lag (OpenAI key roll)
- Plan JSON malformed (Claude synthesis hallucination) — handle via Zod schema validation + retry-once
- Reviewer disagreement (Gemini APPROVE, Codex BLOCK) → MRPF disagreement procedure: SECURITY-labeled CRITICAL → blocking; else recorded synthesis + user notification with both positions
- Token budget exceeded mid-review (plan grows during integration) → truncate plan back to schema-allowed sizes; flag in synthesis

## 12. Risks called out for reviewer scrutiny

1. **Reviewer-loop adversarial scenario** (Gemini v1 CRITICAL 2 + Codex v2 MAJOR 2 — operational rubric). If the synthesizing Claude knows reviewers will critique, it WILL write "safe" minimum-viable plans to maximize APPROVE probability. A prompt-instruction-against this is insufficient — known LLM behavior class.

**Mitigation: mandatory Persona Depth / Ambition Alignment 0-4 rubric (Codex MAJOR 2 — operational, not vague):**

```text
Persona Depth / Ambition Alignment score:
0 = generic plan; ignores persona and decision context entirely.
1 = mentions persona but keeps generic sources/rubric/outputs.
2 = partially adapts scope or outputs, but not enough for requested depth.
3 = materially adapts sources, comparisons, risk checks, and outputs to persona.
4 = expert-grade plan with domain-specific hypotheses, exclusions, scoring, and failure modes.

Required minimum (by depth_target):
executive    = 2
practitioner = 3
expert       = 4

Any plan that scores BELOW the threshold for its depth_target → REQUEST_CHANGES with Origin: plan-ambition.

Specifically rejected for expert persona:
- generic vendor list (>10 candidates with no exclusions)
- vanilla rubric ("price | quality | support" without domain weighting)
- no risk_flags
- no explicit_exclusions with rationale
- rubric_rationale that doesn't tie to decision_context
```

- Reviewer prompts include this rubric verbatim + negative examples
- Synthesizer prompt informed: "Reviewers WILL score your plan on a 0-4 Persona Depth scale. For an `expert` persona, score must be 4 — hedge-bet plans get rejected. See the rubric template in the reviewer prompt for explicit reject criteria."
- §10 test fixtures must include at least one plan at each rubric score (0,1,2,3,4) for each depth_target → 15 fixtures total
- Telemetry tracks Persona Depth score distribution per APPROVE/REQUEST_CHANGES to detect drift

2. **Reviewer overconfidence.** Gemini Deep Think and Codex have BOTH been wrong in prior MERGE-gate reviews (Gemini paste-mode hallucinations per [[feedback_gemini_paste_truncation_hallucination]]; Codex over-grepping per [[feedback_codex_exec_readonly_blocks_own_verification]]). Two-reviewer doesn't mean two correct. Acknowledge residual error rate.

3. **Cost-creep death spiral.** Each plan revision iteration is ~$3-5. If revisions average 3+ rounds, per-job cost balloons past the run cost itself. The `MAX_REVIEW_ROUNDS` cap is the load-bearing safety. Pushback on default `2`?

4. **Reviewer training-data leak.** Plans contain user research topics. Gemini + OpenAI API ToS allow training on free-tier data; check enterprise/data-control settings before sending sensitive topics. Open question: do any of our current Tesla/ASC-class users have NDA implications?

5. **The right gate?** Is plan-synthesis-then-review the right shape, or should we critique the QUESTIONS BEFORE the user answers them? The latter intercepts upstream — fewer wasted user-answer cycles. But also: questions are interactive, harder to gate without UX pain. We're choosing the asynchronous post-answer point because it isolates the cost-stake (plan synthesis is bounded; worker spawn is the asymmetric one). Reviewer: confirm or push back on this gate-location choice.
